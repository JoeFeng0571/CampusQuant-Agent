# 前端优化报告

日期：2026-03-30
优化范围：认证入口体验优化，不修改市场调用逻辑

## 1. 优化背景

本轮从用户视角先执行注册、登录测试，发现认证模块存在明显认知偏差：
- 用户容易误以为系统启用了邮箱验证码或激活邮件
- 已注册邮箱再次注册后没有后续引导
- 宝塔反向代理部署下，固定 IP 写法不利于稳定访问

## 2. 已完成优化

文件：`auth.html`

优化项：
- 增加认证能力说明，明确当前版本未启用邮箱验证码、激活邮件和自助找回密码
- 在登录表单增加说明，明确仅支持“邮箱 + 密码”
- 在注册表单增加说明，明确不会发送验证码，也不会校验邮箱是否真实可用
- 当注册返回“邮箱已被注册”时，自动切换到登录页并预填邮箱
- 认证页 API 地址改为：
  - 本地开发：`http://127.0.0.1:8000`
  - 线上部署：`window.location.origin`

## 3. 优化效果

- 用户能更快理解当前产品真实能力，不再把问题误判为“宝塔邮箱没接通”
- 对“邮箱已注册”场景给出了下一步动作，减少死路
- 认证页在域名反代环境下更容易直接可用

## 4. 仍需后续建设

- 后端未实现邮箱验证码 / 激活邮件
- 后端未实现找回密码 / 重置密码
- 若要真正支持邮箱验证登录，需要新增：
  - 邮件发送能力
  - 验证码或激活 token 存储
  - 验证状态字段
  - 重置密码用户流

## 5. 结论

当前认证模块的核心问题不是“验证码没发出来”，而是产品实际上还没有邮箱验证体系。
本轮优化已把这个事实在前端说清楚，并改善了重复邮箱注册时的操作路径。
如果下一阶段要继续提升交付质量，优先级最高的是补齐密码找回和邮箱验证闭环。
