## 四、CampusQuant → 岗位深度映射

### 4.1 风控系统与 AI 置信度机制 ↔ 广告审核系统

这是最强的类比，**一定要主动讲出来，且要讲到置信度这一层**。

#### 四重风控与审核分级的对应关系

| CampusQuant 风控层 | 触发条件 | 处理结果 | 广告审核类比 |
|-------------------|---------|---------|------------|
| ATR 波动率阻断 | ATR% > 8% | 强制 REJECTED，仓位归零 | 明确违规内容 → 直接拒绝，不进人工 |
| ATR 条件降级 | ATR% 5-8% | CONDITIONAL，仓位减半 | 边缘内容 → 条件通过，需人工复核 |
| 差异化仓位上限 | A股 >15% / 港美股 >10% | 代码强制截断至上限 | 不同市场/品类/地区的差异化审核红线 |
| 单次亏损硬上限 | 潜在亏损 > 3000元 | 反算最大安全仓位后截断 | 单账户/单日投放金额上限保护 |

#### AI 置信度惩罚机制 ↔ 模型置信度分流

| CampusQuant 置信度机制 | 广告审核中的等价设计 |
|-----------------------|-------------------|
| 置信度 < 0.40 → 强制 HOLD，仓位归零 | 模型置信度低于阈值 → 强制转人工，不允许自动放行 |
| 置信度 0.40-0.55 → 线性缩仓（置信度越低仓位越小） | 低置信区间 → 缩小自动审核通过范围，更多送人工 |
| 置信度 ≥ 0.55 → 正常执行，仓位按风控参数计算 | 高置信度 → 自动放行，降低人工审核成本 |
| 多路 Agent 加权融合置信度（基本面×0.35+技术面×0.40+舆情×0.25） | 多维度审核信号加权融合（文本模型+图像模型+账户风险分） |

**面试时的表达**：
> "我识别到一个很关键的风险点：大模型输出的置信度本身是不稳定的，如果直接用置信度来决定要不要执行，会很危险。所以我设计了一套惩罚机制：置信度不够高的时候，不是直接否决，而是把它映射成仓位约束——置信度越低，允许的操作幅度越小，置信度低于绝对底线时才强制 HOLD。这样既保留了模型有价值的判断，又对不确定性做了定量管控。
>
> 这和广告审核里的置信度分流逻辑完全一致：高置信度的判断直接自动处理，低置信度的送人工，中间地带根据置信度区间决定处理方式。区别只是业务领域不同。"

#### 规则层兜底 ↔ 代码层投影

**面试关键句**：
> "我在 CampusQuant 里解决的核心问题，就是'如何让不确定的大模型输出，满足确定性的合规约束'。广告审核本质上是同一个问题——模型会犯错，但违规广告上线的代价是零容忍的，所以一定要在模型之上加确定性的规则投影层，而不是只靠 Prompt 告诉模型'不要违规'。"

**追问"你怎么知道规则层设计对了"**：
> "我跑了对照实验：用没有代码层风控的单智能体 baseline，和有完整风控层的 CampusQuant 对比，baseline 有 30% 的输出违反了至少一项硬约束（仓位超限、止损设置缺失、输出字段不完整），而加了规则投影层之后违规率归零。这证明光靠 Prompt 约束是不够的。"

### 4.2 域感知 RAG 知识检索系统 ↔ 多模块专项知识库

#### 知识覆盖范围

RAG 不是只用于"查股票"，而是一套垂直域知识基础设施：

| 知识类型 | 内容 | 技术实现 |
|---------|------|---------|
| 宏观政策 | 美联储货币政策、中国央行政策、降准降息影响 | 内置种子文档 |
| 行业分析 | A股高景气行业、新能源/科技/消费赛道分析 | 内置种子文档 |
| 风险教育 | 大学生防金融诈骗、常见投资陷阱识别 | 内置种子文档 |
| 外部研报 | 任意 PDF/TXT/Markdown 研报，放入目录自动索引 | `build_kb.py` 离线建库 |
| 实时资讯 | 今日突发新闻、最新财报、政策突变 | DuckDuckGo 实时联网 |

**动静分离架构**：`build_kb.py` 离线预构建 Chroma 向量库和 BM25 倒排索引，在线端直接读取磁盘产物，启动耗时从 ~60s 降至 <2s。

#### 域感知 RAG 路由

每个 Agent 根据自己的分析域独立调用 `search_knowledge_base`，使用专属 query，`max_length` 参数控制截断深度：

| 节点 | 专项 Query | max_length |
|------|-----------|-----------|
| `fundamental_node` | `{symbol} 财务报表 基本面 盈利 机构评级` | 1200 |
| `technical_node` | `{symbol} 近期资金面 行业技术利好利空` | 1000 |
| `sentiment_node` | `{symbol} 最新宏观政策 行业动态 突发新闻` | 1000 |
| `debate_node` | `{symbol} 行业核心风险点 前景 护城河` | 1200 |

`search_knowledge_base(query, market_type, max_length=1500)` 是统一工具层入口，每次调用内部走 BM25+Chroma+DuckDuckGo 三路融合，`max_length` 在工具层截断，节点内无硬编码切片。

**与广告审核的类比**：
> "广告审核里不同检测模块查不同专项知识库——文本安全模块查违禁词政策，图像安全模块查视觉违规分类，落地页模块查恶意域名库。我的设计是同一个思路：基本面 Agent 的检索域是财报/评级，技术面 Agent 的检索域是资金面动向，辩论裁决 Agent 的检索域是行业结构性风险与护城河，每个模块查自己真正需要的知识，而不是共享一份通用上下文。"

#### 混合检索与 RRF 融合

| 检索路 | 优势 | 局限 |
|-------|------|------|
| BM25（稀疏） | 精确匹配代码/机构名（"600519"、"贝莱德"） | 无法捕获语义相似性 |
| Chroma（稠密向量） | 语义召回（"英伟达"≈"NVDA"的 embedding 接近） | 可能漏掉精确关键词 |
| DuckDuckGo（实时） | 覆盖知识库时效盲区（今日突发事件） | 结果质量不稳定 |

RRF 融合公式：`score(d) = Σ 1/(rank_k(d) + 60)`，同时命中 BM25 和 Chroma 的文档得分约为单路 top-1 的 **2倍**（double-hit 效应），无需任何训练。

### 4.3 多智能体架构 ↔ 审核流水线自动化

**DAG 拓扑**（按执行顺序）：

```
data_node → [fundamental / technical / sentiment / rag] 并行 → portfolio_node
                                                              ↓ (has_conflict=True)
                                                         debate_node（≤2轮）
                                                              ↓
                                                         risk_node
                                                              ↓ (risk_rejected, ≤2次重试)
                                                         trade_executor
```

- 四路并行 Agent → 类比广告审核的多维度并行检测（文本/图片/落地页/账户历史）
- 有分歧 → 触发多空辩论仲裁（≤2轮），辩论节点带专项 RAG 作为客观证据层 → 类比审核系统人机协作升级机制
- 风控拒绝 → 触发重试回路（≤2次）→ 类比审核补充材料后重新提审流程

**关键表达**：
> "我理解字节的审核系统不是一个单一模型，而是一个协同的多模块流水线。我在 CampusQuant 里做的事，就是把'如何编排多个 Agent 协同工作、如何处理它们的分歧、如何在任意节点失败时保证系统不崩溃'这些问题系统性地解决了一遍。辩论节点有意思的地方在于，它会先检索一次行业风险与护城河的外部文献，作为裁判的客观依据，而不是只看两个 Agent 各自的逻辑——这和审核员裁决边缘案例时要参考政策手册是同一个思路。"

### 4.4 数据分析能力 ↔ 业务指标跟踪与归因

- **D1-D4 评估协议**：四维量化评估，加权公式 `Acc_w = 0.20×D1 + 0.30×D2 + 0.30×D3 + 0.20×D4`，50只股票测试集，综合准确率 90%
- **指标波动归因**：数据接口出问题时，分层判断是超时（socket 死锁）、代理拦截还是数据源本身故障——这是指标异常归因的实战经验
- **参数调优的 A/B 思维**：移除 Prompt 层置信度限制前后 HOLD 率的变化（~62% → ~41%），本质是一次受控对照实验；代码层 BUY 阈值（0.60）保持不变

## 五、STAR 案例库

### Case 1：解决系统性 HOLD 偏差——AI 置信度机制设计（推荐作为"解决困难"案例）

**Situation**：系统上线后推荐结果中 HOLD 占比异常高（>60%），用户反馈系统"永远不敢给建议"，实际价值大打折扣。

**Task**：找到根本原因并修复——不能简单粗暴改阈值，要在保留风控底线的前提下消除偏差。

**Action**：
1. 排查发现三个叠加原因：① Prompt 中硬写了"置信度 < 0.60 → 强制 HOLD"；② 加权评分 BUY 阈值（0.60）在经验分布下较难触达，HOLD 区间覆盖约 62% 的概率质量；③ portfolio 节点系统 Prompt 有类似软规则产生叠加效果
2. 分层修复：删除 Prompt 中的强制置信度 HOLD 规则（代码层加权评分阈值 `>= 0.60` → BUY、`<= 0.35` → SELL 保持不变，不做下调）→ **保留置信度 < 0.40 时的强制 HOLD**（这是真正需要保护的风控底线，不是偏差）
3. 设计置信度惩罚带（0.40-0.55）：不是直接 HOLD，而是线性缩仓——置信度越低，允许的仓位越小，把不确定性映射为量化约束，消除悬崖效应
4. 固定种子跑对照实验验证修复效果

**Result**：HOLD 区间从理论覆盖 ~62% 的分布压降至 ~41%，系统有效推荐率显著提升，风控底线（< 0.40 强制 HOLD）完整保留，零违规。

**追问准备**：
- "你为什么不直接调低 BUY 阈值？" → 加权评分阈值（0.60）是信号质量的分界线，不是偏差来源；真正的偏差来自 Prompt 层的重复限制和置信度悬崖效应。调低阈值会放进更多低质量信号，惩罚带才是精准解法
- "为什么不去掉所有置信度约束？" → 极低置信度时强制 HOLD 是设计，不是缺陷；置信度 < 0.40 意味着四路信号高度矛盾，此时执行任何操作风险都不可控
- "这和广告审核有什么关系？" → 完全同构：高置信度内容自动放行，低置信度强制人工，中间段根据置信度决定处理方式

### Case 2：混合 RAG 检索精度优化（推荐作为"数据驱动决策"案例）

**Situation**：早期 RAG 只用 BM25 单路检索，对"英伟达"这种中英文混用的金融查询效果差，BM25 依赖字面匹配，"英伟达"和"NVDA"对它来说是完全不同的词。

**Task**：在不引入复杂训练、不增加标注成本的前提下提升检索精度。

**Action**：
1. 引入 Chroma 稠密向量检索，捕获语义相似性（"英伟达"和"NVDA"的 embedding 空间接近）
2. 加入 DuckDuckGo 实时 Web 检索，补充知识库的时效盲区（突发新闻、最新财报）
3. 采用 RRF 无参数融合三路结果：`score(d) = Σ 1/(rank+60)`，BM25 和 Chroma 双命中的文档自动获得约 2× 的得分加成，无需训练
4. 设计"动静分离"架构：`build_kb.py` 离线预构建索引（Chroma 向量库 + BM25 序列化文件），在线端直接加载磁盘产物，启动时间从 ~60s 降至 <2s
5. 设计评估集，量化测试 Recall@5

**Result**：混合 RAG 的 Recall@5 达 88%，较 BM25 单路提升约 24pp；启动性能提升 30×；对"NVDA/英伟达"类跨语言查询效果尤其显著。

**追问准备**：
- "Recall@5 怎么测的？" → 构建查询-相关文档对的评估集，对每个查询取检索结果 Top-5，计算相关文档的召回率
- "为什么用 RRF 不用学习融合？" → 没有足够的标注数据训练融合权重；RRF 是 learning-free 的，鲁棒性强
- "知识库内容是什么？" → 覆盖宏观政策（美联储/央行）、行业分析、风险教育内置种子文档，以及 `data/docs/` 目录下任意 PDF/TXT 研报

### Case 3：攻克代理环境下的数据获取崩溃（推荐作为"技术挑战/抗压"案例）

**Situation**：本地开发时数据接口全部正常，但在 VPN/公司网络环境下，akshare 和 yfinance 的请求会随机无限期 hang 住，整个服务卡死无响应。

**Task**：不换数据源，让系统在任何网络环境下都能稳定运行。

**Action**：
1. 定位根因：TUN 模式代理拦截对东方财富、新浪 CDN 的直连请求，导致 socket 握手永久阻塞
2. 治标：在 `server.py` 启动时注入 `NO_PROXY` 环境变量，将数据源域名加入代理白名单
3. 治本：所有数据请求改为独立线程执行 + `event.wait(timeout)` 非阻塞等待，超时后立即降级至日线收盘价兜底，主线程不阻塞
4. 对港股额外设计 yfinance 主力 + akshare 双路并发，谁先返回用谁

**Result**：系统在代理/非代理/VPN 环境下均可稳定运行，超时降级有日志追踪，用户侧无感知。

### Case 4：域感知 RAG 路由设计（推荐作为"系统设计/主动改进"案例）

**Situation**：做 Code Review 时发现，如果让所有 Agent 共享同一份 RAG 上下文，不同分析域的 Agent 会收到和自己任务不相关的信息——基本面 Agent 需要财报数据，技术面 Agent 需要资金面动向，辩论裁决 Agent 需要行业结构性风险，三者的信息需求完全不同。

**Task**：设计一套让每个 Agent 只获取自己分析域相关知识的 RAG 架构，同时保持工具接口统一、节点间互不干扰。

**Action**：
1. 在 `search_knowledge_base` 工具函数中新增 `max_length: int = 1500` 参数，将截断控制权收回工具层，不在业务代码里硬切字符串
2. 每个节点在函数内部自行调用 `search_knowledge_base`，使用专属 query：
   - `fundamental_node`：`"{symbol} 财务报表 基本面 盈利 机构评级"`，max_length=1200
   - `technical_node`：`"{symbol} 近期资金面 行业技术利好利空"`，max_length=1000
   - `sentiment_node`：`"{symbol} 最新宏观政策 行业动态 突发新闻"`，max_length=1000
   - `debate_node`：`"{symbol} 行业核心风险点 前景 护城河"`，max_length=1200，注入 Prompt 时标注为`【外部研报与宏观事实（作为裁判裁决依据）】`
3. 每个调用都有 `try/except + logger.warning` 降级，单节点 RAG 失败不阻塞整个 DAG
4. 不改动 LangGraph 状态 schema，改动全部封装在各节点函数内部

**Result**：四个节点各查各的专项知识，每节点可用上下文深度 1000-1200 字符，辩论节点有了独立的行业研报依据。改动无 breaking change，旧 `rag_node` 继续作为全局宏观背景节点运行。

**追问准备**：
- "为什么不直接把 rag_context 状态字段传给 debate_node？" → 状态字段里的 rag_context 是用通用 query 检索的结果，debate 需要的是'行业结构性风险和护城河'这类中观内容，专项 query 召回的文档完全不同，混用会稀释信息质量
- "这和广告审核有什么关系？" → 广告审核的多模块流水线也是同一个问题：文本安全模块查违禁词政策，图像模块查视觉违规分类，落地页模块查恶意域名——每个模块查自己的专项库，而不是共享一份通用上下文

## 六、高频面试问题 & 答题要点

### 产品/业务类

**Q: 你理解国际化商业安全团队在做什么？**

核心张力是"**安全 vs 增长**"——过度审核会误杀正常广告（影响商业收入），审核不足会产生违规（破坏用户信任和合规）。这和 CampusQuant 的"过度保守（HOLD偏差）vs 风控底线"是完全同构的设计问题。团队的价值在于把这条线划在最优位置，而不是一味收紧。

**Q: 如果你发现某个核心广告指标突然下降30%，你怎么排查？**

1. 先验证数据准确性（口径、时区、采集链路是否故障）
2. 拆分维度：按市场/平台/广告类型/时间段分层，定位范围
3. 外部因素：节假日、政策变化、竞品动作
4. 内部变更：审核规则上线、产品改动、模型迭代
5. 输出归因结论 + 建议行动

类比 CampusQuant：数据接口异常时我也是这个排查逻辑——先判断是数据层问题（超时/代理）还是业务逻辑问题，再按层次修复。

**Q: 如何平衡审核严格度和广告主体验？**

这是精准率 vs 召回率的经典权衡。关键在于**分层设计**而不是单一阈值——类比 CampusQuant：ATR>8% 是硬拒绝（高置信违规），5-8% 是 CONDITIONAL（人工介入），<5% 放行。置信度也分三档处理，而不是二元判断。同时要有反馈机制（误判申诉）持续校准阈值。

### Agent/大模型类（HR 特别强调）

**Q: 你做的 Agent 系统和普通调 API 的区别是什么？**

普通调 API 是单轮、单模型、无状态的。Agent 系统的核心差异在于：多步骤规划、工具调用、状态管理、多 Agent 协同，以及最重要的——**错误处理、降级和回路控制**。CampusQuant 里每个节点都有 fallback，整个 DAG 有重试回路和反向路由，这才是生产可用的 Agent，而不是 demo。

**Q: 大模型输出不可控，你怎么保证系统可靠性？**

三层防御：
1. **Pydantic 强制结构化输出**：模型输出必须符合预定义 schema，字段类型和值域在进入业务逻辑之前就已校验
2. **代码层风控投影**：即使 LLM 输出了违规仓位，代码强制截断到合规范围，LLM 无法绕过
3. **节点降级**：LLM 调用失败时返回默认保守输出（HOLD，仓位归零），不崩溃，不阻塞

这套防御体系的设计原则是：**不信任 LLM 的任何输出，只信任经过代码验证的输出**。

**Q: 你的 RAG 是怎么保证每个 Agent 拿到的知识是相关的？**

> "每个 Agent 自己构造专领域 query 发起检索：基本面查'财务报表/盈利/机构评级'，技术面查'行业资金面/技术利好利空'，辩论节点查'行业核心风险点/护城河'。query 不同，从 BM25+Chroma 里召回的文档就完全不同。`search_knowledge_base` 有 `max_length` 参数控制截断深度，各节点按需配置，基本面和辩论节点用 1200，技术面和舆情用 1000，做到既够用又不撑大 Prompt。"

**Q: 你觉得大模型在广告审核场景中的局限是什么？**

- **一致性**：同一条广告不同 temperature 下结论可能不同，难以保证审核一致性
- **置信度校准差**：模型对明显违规内容可能给出高置信通过，对边缘内容反而给低置信——置信度不能直接用来做决策
- **不可解释**：很难向广告主解释"为什么被拒了"，申诉处理成本高
- 因此：**模型负责召回，规则负责兜底，人工负责边界案例**——这是我在 CampusQuant 里验证过的分层架构

### 10.1 置信度是怎么算出来的（三阶段链路）

#### 第一阶段：每个分析师 LLM 自己输出原始 confidence

代码位置：`nodes.py` `_ANALYST_REPORT_SKELETON` Prompt

LLM 被强制按以下规则计算，不允许输出 0.60/0.65 这类"敷衍默认值"：

```
基础分：0.50（完全中性信号）
动态加减（可叠加）：
  多项核心指标高度一致且强烈看涨/看跌  →  +0.25 ~ +0.35
  单一利好/利空信号明确               →  +0.10 ~ +0.20
  信号轻微矛盾或噪音                   →  -0.05 ~ -0.10
  核心数据极度恶化或极强反转信号        →  +0.30 ~ +0.40
  信号严重矛盾或数据不足               →  -0.15 ~ -0.25
```

示例：基础 0.50 + 技术金叉 +0.15 + 业绩超预期 +0.12 = **confidence = 0.77**

#### 第二阶段：`_compute_weighted_score` 将三路融合为综合分

代码位置：`nodes.py`（portfolio_node 内）

```python
# 建议→分值（BUY=1.0, HOLD=0.5, SELL=0.0）
_REC_SCORE = {"BUY": 1.0, "HOLD": 0.5, "SELL": 0.0}

# 各路：信号分 = 建议分值 × 置信度
f_score = _REC_SCORE[f_rec] * f_conf
t_score = _REC_SCORE[t_rec] * t_conf
s_score = _REC_SCORE[s_rec] * s_conf

# 按市场加权（A股: w_f=0.35, w_t=0.40, w_s=0.25）
weighted_score = w_f*f_score + w_t*t_score + w_s*s_score  # 综合方向分
avg_confidence = w_f*f_conf  + w_t*t_conf  + w_s*s_conf   # 综合置信度

# 方向阈值
BUY  if weighted_score >= 0.60
SELL if weighted_score <= 0.35
HOLD otherwise
```

**数字示例（A股，三路均看涨）：**

| 分析师 | 建议 | 置信度 | 权重 | 贡献分 |
|--------|------|--------|------|--------|
| 基本面 | BUY  | 0.78 | 0.35 | 1.0×0.78×0.35 = 0.273 |
| 技术面 | BUY  | 0.82 | 0.40 | 1.0×0.82×0.40 = 0.328 |
| 舆情   | BUY  | 0.65 | 0.25 | 1.0×0.65×0.25 = 0.163 |
| **合计** | — | — | — | **weighted_score = 0.764 → BUY** |

avg_confidence = 0.78×0.35 + 0.82×0.40 + 0.65×0.25 = **0.763**

#### 第三阶段：`_apply_confidence_penalty` 将 avg_confidence 映射为仓位约束

代码位置：`nodes.py`（`_apply_confidence_penalty` 函数）

```python
_CONF_FLOOR     = 0.40   # 低于此值强制 HOLD
_CONF_THRESHOLD = 0.55   # 低于此值线性缩仓

if confidence < 0.40:
    return "HOLD", 0.0                           # 强制 HOLD，仓位归零

if confidence < 0.55:
    scale = (confidence - 0.40) / (0.55 - 0.40) # 线性 0~1
    penalized = base_pct * scale
    return action, penalized                     # 保留方向，缩减仓位

return action, base_pct                          # 正常执行
```

**线性缩仓数字示例（base_pct=10%）：**

| avg_confidence | scale 计算 | 实际仓位 | 说明 |
|---------------|-----------|---------|------|
| 0.35 | — | **0%** | 强制 HOLD |
| 0.45 | (0.45-0.40)/0.15 = 0.33 | **3.3%** | 缩至三分之一 |
| 0.50 | (0.50-0.40)/0.15 = 0.67 | **6.7%** | 缩至三分之二 |
| 0.60 | — | **10%** | 正常执行 |

**面试一句话解释**：
> "置信度不是一个单一数，是三个专家各自置信度按市场权重的加权平均。它最终不决定买还是卖，而是决定买多少——置信度越低仓位越小，而不是非此即彼地强制 HOLD。这和广告审核里置信度决定自动处理比例的逻辑完全一致。"

### 10.2 综合准确率 90%（四维评测流水线）

**加权公式**（`eval_pipeline.py`）：

```python
Acc_w = 0.20×D1 + 0.30×D2 + 0.30×D3 + 0.20×D4
```

**四个维度的判定标准：**

| 维度 | 权重 | 通过标准 |
|------|------|---------|
| D1 市场分类准确率 | 20% | `MarketClassifier.classify()` 输出与预设市场匹配 |
| D2 数据获取成功率 | 30% | 成功拉取 ≥5 日 OHLCV，最新价 >0 |
| D3 研报结构完整性 | 30% | trade_order≠None + simulated=True + rationale≥30字 + ≥2/3分析师报告完成 |
| D4 风控合规率 | 20% | 仓位≤上限(A股15%/港美10%) + 止损已设 + 止损比例≥5%（全市场统一下限） |

**90% 的来源：**
- D1=D2=**100%**（50只全通过：D1靠三级分类兜底，D2靠双路并发+日线降级保障）
- D3≈D4≈**83%**（少量因LLM超时或输出格式不规范未通过）
- 代入公式：0.20×1.0 + 0.30×1.0 + 0.30×0.83 + 0.20×0.83 = **0.92 ≈ 90%**

**追问"D1/D2为什么能100%"**：D1 靠正则+200条字典+新浪API三级匹配，50只测试集代码格式均清晰；D2 靠双路并发+超时降级，保证"有数据返回"而不是直接报错。

### 10.3 混合 Recall@5 = 88%，较 BM25 单路提升约 24pp

**Recall@5 的含义**：对每个查询，取检索返回的 Top-5 文档，计算其中包含相关文档的比例。88% 意味着 100 个查询中 88 个能在 Top-5 找到相关文档。

**两套系统对比：**

| 系统 | Recall@5 | 说明 |
|------|---------|------|
| BM25 单路 | ~64% | 依赖字面匹配，"英伟达"和"NVDA"对它是完全不同的词 |
| 混合 RAG（BM25+Chroma+RRF） | ~88% | Chroma 捕获语义相似性，RRF double-hit 加权 |
| **提升** | **+24pp** | |

**RRF 为什么有效（double-hit 数字）：**

查询"英伟达"时：
- BM25：召回含"英伟达"的文档，rank=1，贡献分 = 1/(1+60) = **0.0164**
- Chroma：召回含"NVDA"的文档（embedding 空间接近），含"英伟达"的文档 rank=2，贡献分 = 1/(2+60) = **0.0161**
- 双命中文档总分 = 0.0164 + 0.0161 = **0.0325**，约为单路 top-1 的 **2倍**，自动排到第一

**追问"评估集怎么构建的"**：
> "选取了覆盖三市场的典型查询，包括公司中文名、英文代码、行业关键词三类，手动标注每个查询对应的相关文档，构成查询-相关文档对。评估集侧重中英文混用场景，因为这是金融垂直域最典型的挑战。"
