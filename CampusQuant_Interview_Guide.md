## 一、项目背景与产品定义

### 简历原文
> 通过200+问卷调查与15场深入访谈，确定大学生有闲钱想投资却不会分析、缺乏风控意识等集中痛点，规划"AI研报+模拟交易+财商教育"三位一体平台。产品层设置风险基线，熔断实盘、杠杆交易。识别AI置信度波动对本金的潜在危害，设计置信度惩罚机制，将模型不确定性映射为仓位约束（置信度<0.40强制HOLD，0.4-0.55线性缩仓），保障大学生本金安全。

### 痛点挖掘与用户洞察（PM视角）

**目标用户画像**：在校大学生，月可支配资金 500–3000 元，对投资有兴趣但缺乏系统性认知，市面上的工具（同花顺、雪球）信息密度过高，学习曲线陡峭，直接开户炒股风险极大。

**三位一体产品架构的逻辑**：

| 模块 | 解决的问题 | 产品价值 |
|------|-----------|---------|
| AI 研报 | 信息不对称、分析能力不足 | 让用户"看懂"一只股票 |
| 模拟交易 | 无实战经验、不敢入场 | 零成本试错，建立手感 |
| 财商教育 | 缺乏系统知识体系 | 从知识到决策的完整闭环 |

三者并非孤立功能，而是一条**认知成长路径**：用户先通过 AI 研报了解标的 → 在模拟盘验证自己的判断 → 通过教育内容修正认知偏差。这是产品路线图（Roadmap）层面的顶层设计，体现了 PM 对用户旅程（User Journey）的完整规划能力。

**风险基线设计（核心产品决策）**：

面向学生群体，若产品放开实盘交易、杠杆、期货等功能，一旦出现亏损将直接损害用户信任，且可能涉及监管合规风险（未成年/在校生金融产品限制）。因此在**产品定义阶段**就做出了"熔断"决策：
- `TradeOrder.simulated` 字段强制为 `True`，架构层面物理隔断实盘通道
- 不接入任何真实券商 API（无 IBKR、无 Binance）
- 模拟撮合引擎完全本地化

这是典型的**产品风控思维**：在功能边界上划红线，而不是依赖用户自律。能在面试中说清楚"为什么不做"，比说清楚"怎么做"更能体现 PM 成熟度。

**置信度惩罚机制（AI不确定性的产品化处理）**：

这是本项目最具差异化的产品决策之一，体现了 PM 对 AI 系统局限性的深度理解。

**核心洞察**：LLM 输出的置信度并非装饰性数字——当模型自身对分析结论把握不足时（置信度低），强行给出 BUY/SELL 建议会产生均值化偏差，即模型倾向于给出"看起来合理"但实际上没有强力依据的建议。对于没有分析能力的大学生用户，这种不确定性的危害比明显错误更大（错误可识别，不确定性不可感知）。

**机制设计**（直接映射到代码实现）：

```
置信度 < 0.40  → 强制 HOLD，不论分析师建议是什么
0.40 ≤ 置信度 < 0.55  → 线性缩仓：quantity_pct × (confidence / 0.55)
置信度 ≥ 0.55  → 正常执行，不干预
```

**为什么是 0.40 和 0.55 这两个阈值？**
- 0.40 是"分析毫无把握"的底线——低于此值意味着模型自己都认为有 60% 的概率分析有误，此时任何仓位建议都是噪声
- 0.55 是"基本有依据"的门槛——适度线性过渡，避免阈值处的悬崖效应（0.39 全仓，0.40 清仓）
- 两个值之间的线性区间保证了机制的平滑性，不产生突变

**产品意义**：置信度惩罚机制把模型的内部状态（不确定性）变成了用户可受益的保护机制，是 AI 产品"以用户利益为中心设计"的具体体现。

## 二、多Agent架构与调度

### 简历原文
> 基于LangGraph设计8节点DAG工作流，支持基本面/技术面/舆情/RAG四路并行分析，冲突时自动触发多空辩论路由（≤2轮）；设计A/港/美股差异化推理Prompt与防死循环机制；基于FastAPI+SSE实现前端流式交互，实现推理过程全程流式可视化，AI决策白盒透明可追溯。

### LangGraph 与 DAG 工作流（后端视角）

**为什么用 LangGraph，而不是简单的链式调用？**

普通 LangChain Chain 是线性的：输入 → 处理 → 输出。但金融研报生成需要：
1. **并行执行**：基本面分析、技术面分析、舆情分析三者互相独立，串行执行浪费时间
2. **条件路由**：风控节点根据风险评分决定是否进入辩论轮次，不能是固定流程
3. **状态共享**：所有节点读写同一个 `TradingGraphState`（TypedDict），节点间解耦但数据互通
4. **循环控制**：辩论节点需要最多迭代 2 轮，必须有终止条件

LangGraph 的 `StateGraph` 天然支持以上四点，这是选型决策，而非随意使用某个框架。

**8 节点 DAG 的具体拓扑**：

```
START
  └─► data_node（数据拉取、标准化、市场分类）
        ├─► fundamental_node  ┐
        ├─► technical_node    ├─ 并行执行（Fan-out）
        ├─► sentiment_node    │
        └─► rag_node          ┘
              └─► debate_node（汇总并行结果 + 多空辩论，≤2轮）
                    └─► risk_node（四重风控决策，≤2次重试）
                          └─► trade_executor（模拟下单 + 置信度惩罚 + 极值阻断）
                                └─► END
```

**并行分析（Fan-out/Fan-in 模式）**：

`fundamental/technical/sentiment/rag` 四个节点在 `data_node` 完成后同时启动，各自独立调用 LLM，结果写入 `TradingGraphState` 的不同字段。`debate_node` 作为 Fan-in 节点，先汇总四路分析结果，判断是否存在分歧，再决定是否进入辩论轮次。这在 LangGraph 中通过并行 edge 配置实现，避免了手工管理 `asyncio.gather`。

**辩论路由（条件边）**：

```python
# 伪代码示意
def debate_router(state):
    if state["debate_rounds"] >= MAX_DEBATE_ROUNDS:  # MAX=2
        return "risk_node"      # 超出轮次，强制结束
    if not state["has_conflict"]:
        return "risk_node"      # 无分歧，直接进入风控
    return "debate_node"        # 继续辩论
```

这是 PM 需求"避免无限循环"到后端实现的完整链路——需求侧是"研报不能卡死"，实现侧是 `debate_rounds` 计数器 + `has_conflict` 标志位 + 条件边。

**A/港/美股差异化推理 Prompt（Prompt 工程）**：

不同市场的信息结构差异显著：
- **A股**：散户主导，舆情（消息面）权重更高；政策风险显著，需强化基本面的政策敏感度分析
- **港股**：机构主导，技术面与南向资金流向权重更高
- **美股**：财报披露严格，基本面（EPS、营收增速）权重最高，舆情权重相对较低

差异化体现在 `_PROMPTS` 字典的 Prompt 工程层面——不同市场的分析维度指令强度不同，而非仅仅切换数据源。所有 Prompt 集中存储在 `nodes.py` 的 `_PROMPTS` 字典中，方便统一维护和迭代。

**防死循环机制（Anti-Loop）**：

```python
# TradingGraphState 中
tool_call_counts: Annotated[Dict[str, int], _merge_counts]
MAX_TOOL_CALLS = 3

def _check_tool_limit(state, node_name):
    if state["tool_call_counts"].get(node_name, 0) >= MAX_TOOL_CALLS:
        return True  # 触发熔断，跳过工具调用
```

`_merge_counts` 是并行安全的自定义 reducer（dict.update 语义），保证多个并行节点各自写入自己的计数器 key 时不会相互覆盖（LangGraph 默认的 last-write-wins 语义会导致并行节点计数丢失）。这是参考 TradingAgents-CN-main 开源项目架构改进的关键点，面试中可主动提及"发现了开源代码的并发 Bug 并修复"。

**FastAPI + SSE 流式可视化（后端+前端视角）**：

**为什么用 SSE 而不是 WebSocket？**

| 对比维度 | SSE | WebSocket |
|---------|-----|-----------|
| 通信方向 | 服务端 → 客户端（单向） | 双向 |
| 协议 | HTTP/1.1（标准，无需升级） | WS（需握手） |
| 实现复杂度 | 低（`sse-starlette`，几行代码） | 高 |
| 适用场景 | 服务端推送进度、日志流 | 实时双向交互 |

研报生成是**服务端推送进度**的典型场景——用户发出请求后只需接收结果，不需要中途发消息。SSE 比 WebSocket 更轻量，且天然支持 HTTP 缓存、负载均衡，是正确的技术选型。

**9 种 SSE 事件类型的设计**：

```
start → node_start → node_complete → conflict → debate
→ risk_check → risk_retry → trade_order → complete / error
```

每个事件类型对应前端 DOM 的不同渲染逻辑：`node_start` 触发进度条动画、`debate` 触发多空观点的动态打字效果、`trade_order` 触发结果卡片展示。这是**事件驱动 UI**的设计模式，将后端 Agent 状态变更与前端视觉反馈精确对齐。

**白盒化的产品意义**：

传统 AI 产品给出结论但不解释过程（黑盒），用户无法建立信任。CampusQuant 的 SSE 流式推送让用户实时看到：
- 哪个 Agent 正在工作
- 多空双方各自的论据是什么
- 风控模块为何给出特定仓位建议

这对于**学生用户的财商教育**价值极高——研报生成过程本身就是一次 AI 辅助的投资分析教学。

## 三、异构数据管道与混合RAG

### 简历原文
> 统一封装A/港/美股多源API，配套内存缓存与指数退避重试容错机制，实现稳定数据获取。搭建混合RAG（Chroma稠密向量+BM25稀疏关键词+实时Web检索引擎），结合RRF重排序融合多路结果，混合Recall@5达88%，较BM25单路提升约24pp，提升垂直域金融研报的检索精度。

### 异构数据管道（后端视角）

**为什么说"异构"？**

三个市场的数据源完全不同，字段格式、接口协议、可靠性都存在显著差异：

| 市场 | 主要数据源 | 问题 |
|------|-----------|------|
| A股 | akshare `stock_zh_a_spot_em` | 频繁 RemoteDisconnected，字段顺序不直觉 |
| 港股 | akshare `stock_hk_spot_em` | 接口超时率高 |
| 美股 | yfinance | 稳定，但需处理时区与货币单位 |
| A股指数 | 新浪财经直连（主）+ akshare（备） | 东方财富 EM 接口在代理环境下被拦截 |
| 全球指数 | yfinance（主）+ akshare 全球表（备） | yfinance 港美指数最稳定 |

统一封装的核心价值是：**上层 Agent 节点不需要知道数据从哪里来**。所有数据经过 `DataLoader._standardize()` 后，统一输出 `[timestamp, open, high, low, close, volume]` 标准格式，节点只消费标准化数据。

**内存缓存（TTL 300s）**：

市场行情页面用户频繁刷新，如果每次都请求外部 API，既慢又容易触发限流。TTL 缓存将同一标的 5 分钟内的重复请求全部命中本地内存，首次冷请求 10s+，缓存命中后降至 100ms 以内。

**指数退避重试（Exponential Backoff）**：

```python
# 逻辑示意
for attempt in range(3):
    try:
        return fetch_data(symbol)
    except (RemoteDisconnected, Timeout):
        wait = 1.5 ** attempt      # 1s → 1.5s → 2.25s
        time.sleep(wait)
raise DataFetchError("all retries exhausted")
```

退避而非固定间隔的原因：若服务端过载，固定间隔重试会进一步加剧拥塞；指数退避给服务端喘息时间，是标准的分布式系统容错模式。

**大盘指数的 Fail-Fast 容灾体系（实际工程案例）**：

这是项目中一个典型的问题发现 → 诊断 → 修复的完整案例，面试中可作为"遇到的最难工程问题"来讲。

**问题**：后端 `/api/v1/market/indices` 接口频繁出现 12s 以上的响应延迟，伴随 `RemoteDisconnected` 警告。

**诊断**：通过基准测试发现——东方财富 EM 接口（`push2.eastmoney.com`）在国内代理环境下被拦截，`akshare` 内部有 `request_with_retry` 自带重试，外层又有 `_akshare_with_retry(retries=1)`，双重重试导致每次失败要等 **5.7~8.6 秒**才放弃。

**根因**：并非 akshare 代码问题，而是网络环境问题——代理服务器拦截了东方财富 CDN 节点的 HTTPS 请求。

**修复方案**：
1. 外层重试 `retries=1 → 0`（移除外层重试，避免双重等待）
2. Event wait timeout `12s → 4s`（Fail-Fast，4s 后放弃 akshare 结果）
3. 优先级调整：新浪财经直连（hq.sinajs.cn）从"最低优先"提升为"覆盖 akshare"的中高优先
4. 新浪直连覆盖全部 4 个国内指数（000001/000688/399001/399006），稳定性 3/3，平均 619ms

**修复效果**：最坏情况响应时间从 12s → 4s，正常情况由新浪直连在 ~700ms 内完成国内指数数据获取。

这个案例的面试价值在于：展示了从日志警告 → 基准测试 → 根因分析 → 最小化修复的完整工程思路，而不是盲目换接口。

**多源并发 + 优先级合并策略**：

```
并发执行（4路线程）：
  akshare A股表  ──wait 4s──►  (成功则用，失败则跳过)
  akshare 全球表 ──wait 4s──►  (成功则用，失败则跳过)
  新浪直连       ──wait 5s──►  (覆盖 akshare 国内指数，稳定性更高)
  yfinance       ──wait 15s─►  (覆盖全球指数 HSI/NDX/SPX/DJIA)

合并优先级：yfinance > 新浪直连 > akshare
```

这是**电路熔断（Circuit Breaker）**思想：单一数据源故障不影响整体，降级链确保前端永远有数据返回。

### 混合 RAG 架构（后端视角）

**为什么纯向量检索不够？**

金融研报中存在大量专有名词：股票代码（`600519`）、指标缩写（`ROE`、`PE-TTM`）、公司简称（`茅台`、`宁德时代`）。向量模型对这类精确字符串的语义编码能力弱，相似度计算容易"丢失"关键词匹配——用户问"茅台的毛利率"，纯向量检索可能召回"白酒行业综述"而非"贵州茅台财报分析"。

**三层混合检索架构**：

```
用户查询
   ├─► Chroma（稠密向量）  → 语义相近的文档片段（擅长泛化理解）
   ├─► BM25（稀疏关键词）  → 精确匹配专有名词、数字（擅长精确召回）
   └─► DuckDuckGo Web引擎 → 实时新闻、公告（擅长时效性信息）
         │
         ▼
      RRF 重排序（Reciprocal Rank Fusion）
         │
         ▼
      Top-K 文档片段 → 输入 LLM
```

**RRF 重排序算法**：

RRF 不依赖各检索器的原始相关性分数（不同检索器的分数不可比较），而是基于**排名位置**融合：

```
RRF_score(d) = Σ 1 / (k + rank_i(d))
```

文档 d 在多个检索结果中排名越靠前，RRF 分数越高。k=60 是经验超参，平衡头部文档与尾部文档的权重差距。这个算法的优势是无需训练、无需标注数据，适合快速迭代的工程场景。

**量化效果（内测值）**：

在金融垂直域 25 条测试查询（含精确股票代码、财务指标的复杂问题）下：
- 混合 RAG Recall@5：**88%**
- BM25 单路 Recall@5：约 64%
- 混合较单路 BM25 提升约 **24pp**

提升主要来自两类查询：（1）语义泛化查询（向量检索贡献）；（2）含精确数字/代码的查询（BM25 贡献）。RRF 融合后两类都能稳定召回。

## 四、多层风控体系与全栈产品化

### 简历原文
> 代码层实施四重不可豁免风控红线：差异化仓位上限、ATR%波动率阻断、单次亏损硬上限、LLM异常保守降级，有效建立了用户对AI投顾的安全信任。构建Agent四维评测流水线（市场分类/数据获取/研报完整性/风控合规），测试集覆盖A/港/美股50只标的，综合准确率约90%，完成从底层架构到前端交互的完整产品闭环。

### 四重不可豁免风控红线（后端视角）

**"不可豁免"的工程含义**：风控逻辑实现在代码层，无论 LLM 输出什么，都会被代码强制覆盖，不依赖 Prompt 约束（Prompt 可以被绕过，代码不会）。

**四重红线的具体实现**：

**① 差异化仓位上限**
```python
# A股最高 15%，港股/美股最高 10%
position_cap = 15.0 if market_type == "A_STOCK" else 10.0
if order.quantity_pct > position_cap:
    order.quantity_pct = position_cap
```
设计依据：大学生可用资金约 5 万元，15% 对应 7500 元，单笔亏损上限约 750 元（按 10% 止损），在心理可承受范围内。

**② ATR% 波动率阻断**
```python
# ATR% > 8%：直接拒绝，风险过高
# ATR% > 5%：有条件批准，仓位减半
if atr_pct > 8:
    decision.approval_status = "REJECTED"
elif atr_pct > 5:
    decision.position_pct *= 0.5
    decision.approval_status = "CONDITIONAL"
```
ATR（平均真实波动幅度）是衡量标的日内波动性的标准指标。高 ATR% 意味着标的价格剧烈波动，止损线难以设置，学生用户很难在情绪稳定的状态下执行止损。

**③ 单次亏损硬上限**
```python
# 假设本金 5 万，最大亏损不超过 ¥3000
capital = 50000
max_loss = capital * (position_pct / 100) * (stop_loss_pct / 100)
if max_loss > 3000:
    # 按比例缩减仓位，确保亏损 ≤ ¥3000
    position_pct = 3000 / (capital * stop_loss_pct / 100)
```
¥3000 是调研中大学生自报"能接受的最大单次损失"中位数。硬编码为产品安全上限，比用户自律更可靠。

**④ LLM 异常保守降级**
```python
# LLM 输出异常时（Pydantic 验证失败 / 超时 / 幻觉），触发保守降级
fallback_decision = RiskDecision(
    approval_status="REJECTED",
    position_pct=0.0,
    rationale="LLM输出异常，触发保守降级，建议HOLD"
)
```
当 LLM 输出不可信时，宁可"不操作"也不给出错误建议。这是**Fail-Safe**设计原则：系统失效时趋向安全状态（0仓位），而非危险状态（随机下单）。

**置信度惩罚与极值阻断（trade_executor 层）**：

风控红线在 `risk_node` 执行，但 `trade_executor` 还有第二道防线：
- **置信度惩罚**：`confidence < 0.40` 强制 HOLD，`0.40–0.55` 线性缩仓
- **极值阻断**：`quantity_pct > position_cap` 截断，`stop_loss` 超出合理范围时强制修正

两道防线串联，形成纵深防御：即使风控节点有疏漏，执行节点也会兜底。

### Agent 四维评测流水线（PM+后端双重视角）

**为什么要自建评测体系？**

LLM 输出是概率性的，"感觉不错"不是质量保证。在金融场景下，一个错误的分析结论可能直接影响用户决策。评测体系是**产品质量的最后一道防线**，也是持续迭代的基础设施。

**四个评测维度（D1-D4）**：

| 维度 | 评测内容 | Pass 条件 |
|------|---------|-----------|
| D1 市场分类 | 股票代码能否被正确归类为 A/港/美股 | 分类结果与 Ground Truth 一致 |
| D2 数据获取 | 能否成功拉取到有效行情数据（非全0/fallback） | 关键 OHLCV 字段非空非零 |
| D3 研报完整性 | 研报是否包含基本面/技术面/情绪面/结论四个必要模块，各模块是否有实质内容 | 报告数≥3，rationale 字符数≥50 |
| D4 风控合规 | trade_order 中的仓位/止损参数是否符合风控红线 | A股仓位≤15%，港美≤10%；止损≥5%（A股）或7%（港美） |

**测试集覆盖**：A/港/美股各约 17 只，共 50 只标的，涵盖大中小盘、高低波动率标的。

**综合准确率约 90%（内测值，待扩大测试集验证）**：

四个维度加权后得出，其中 D1（市场分类）准确率最高（接近 100%），D2（数据获取）受网络环境影响有少量失败，D3/D4 是核心质量维度，也是改进重点。

**评测体系对产品迭代的价值**：每次修改 Prompt 或风控参数后，跑一遍评测流水线即可量化改动效果，避免"改了这里坏了那里"的盲目迭代。

### Pydantic 约束（后端视角）

**Pydantic 的工程价值**：

LLM 原始输出是字符串，直接解析极易出错。通过定义 6 组 Pydantic 模型，配合 `with_structured_output()`，强制 LLM 按 schema 输出，从源头杜绝字段缺失、类型错误等问题：

| Pydantic 模型 | 对应节点 | 关键约束 |
|--------------|---------|---------|
| `AnalystReport` | fundamental/technical/sentiment | confidence ∈ [0,1]，signal_strength 枚举值 |
| `AnalystReport` | rag_node | 同上 |
| `DebateOutcome` | debate_node | debate_summary 最少 80 字符 |
| `RiskDecision` | risk_node | position_pct ∈ [0,20] |
| `TradeOrder` | trade_executor | rationale 最少 30 字符，simulated 永为 True |

```python
class RiskDecision(BaseModel):
    approval_status: Literal["APPROVED", "CONDITIONAL", "REJECTED"]
    position_pct: float = Field(ge=0, le=20)  # 仓位 0–20%，代码层截断
    stop_loss_pct: float                       # 止损比例
    rejection_reason: Optional[str] = None
```

`Field(ge=0, le=20)` 确保仓位比例永远在合理范围内，不需要在下游节点再做校验。这是**防御性编程**的体现。

## 五、高频面试问题预备

**Q1：你在这个项目里更偏 PM 还是偏后端？**
> 两者都做，且是刻意为之。产品决策（风险基线、置信度惩罚阈值设计、评测指标定义）与工程实现（LangGraph DAG、混合 RAG、SSE 协议、风控红线代码化）由同一个人完成，确保了产品设计与技术实现之间没有信息损耗。这是小团队 Lead 的必备能力，也是我在这个项目中刻意培养的复合能力。

**Q2：8节点 DAG 和普通的 LangChain Chain 有什么本质区别？**
> Chain 是有向无环的线性序列，不支持并行、条件分支和循环。LangGraph 的 StateGraph 支持 Fan-out 并行、条件边（基于状态字段路由）和受控循环（通过计数器终止），是图结构而非链结构。这使得 4 个分析节点可以同时执行，而不是排队等待，是降低响应时间的核心机制。

**Q3：混合 RAG 的 BM25 和向量检索各自的局限性是什么？**
> BM25 是词频统计模型，对语义泛化无能为力——"净利润"和"盈利能力"对 BM25 是两个完全不同的词。向量检索擅长语义相似但无法精确匹配数字和代码。混合后的 RRF 重排序让两者互补，但代价是推理延迟增加（三路并行检索）和 context window 占用增大（更多候选文档）。这是刻意接受的 trade-off，换来的是 Recall@5 从 64% 提升至 88%。

**Q4：90% 综合准确率是怎么测出来的？**
> 构建了四维评测流水线（D1-D4）：市场分类、数据获取、研报完整性、风控合规，测试集覆盖 A/港/美股 50 只标的。四个维度独立评分后加权得出综合准确率约 90%。这是内测值，样本量 50 只是已知局限，后续可扩展到更大规模。重点是评测体系本身的设计——D1-D4 四个维度覆盖了从数据输入到风控输出的完整链路，任何一个环节的问题都能被定位到。

**Q5：置信度 0.40/0.55 的阈值是怎么确定的？**
> 0.40 是"模型自己都不确定"的边界——低于此值意味着模型认为有 60% 概率分析有误，此时任何仓位建议都是噪声，强制 HOLD 是最安全的决策。0.55 是"基本有依据"的门槛，之间用线性区间过渡，避免阈值处的悬崖效应。这两个值是通过对不同阈值下模拟回测结果的主观评估确定的，不是严格统计意义上的最优解，面试中可坦诚这一点，并说明后续可用更大数据集做阈值优化。

**Q6：为什么选择 SQLite 而不是 PostgreSQL？**
> 部署阶段的现实考量：SQLite 零依赖、单文件、无需独立服务，对于 MVP 阶段的校园项目部署成本极低。通过 SQLAlchemy 2.x 异步 ORM + aiosqlite 驱动，上层代码与数据库完全解耦——迁移到 PostgreSQL 只需修改 `DATABASE_URL` 环境变量，无需改动任何业务代码。这是**可扩展性优先于过早优化**的架构决策。

**Q7：遇到的最难的工程问题是什么？**
> 大盘指数接口的 Fail-Fast 修复。现象是 `/api/v1/market/indices` 接口频繁 12s 以上超时，日志里有 `RemoteDisconnected` 警告。排查过程：写基准测试脚本对比 4 个备选接口，发现东方财富 EM 接口在本机代理环境下 0/3 成功率（立即 ProxyError），akshare 内部有自带重试，加上外层重试共 4 次，每次失败等 5-8 秒。新浪财经直连稳定性 3/3、619ms。修复方案：外层重试改为 0 次，Event wait 从 12s 降到 4s，新浪直连优先级提升为"覆盖 akshare"。整个排查到修复不到 2 小时，关键是先测量、再修复，而不是凭感觉换接口。

## 六、V1.2 新增功能深度解析（后端研发视角）

### 6.1 三币种模拟交易账户

**产品决策（PM）**：大学生可用资金以人民币计价，但若用同一账户买港股/美股，RMB 金额会与 HKD/USD 价格混算，造成资产幻觉（如"买了1手腾讯花了370"到底是 370 RMB 还是 370 HKD？）。三币种分账从产品层消除了这个认知误区，同时也是真实券商账户的结构映射，有助于学生建立正确的外汇风险概念。

**工程实现（后端）**：在 `VirtualAccount` 中用 `cash_cnh/cash_hkd/cash_usd` 三个字段代替单一 `cash`，`place_order` 根据 `market_type` 字段路由到对应货币池扣款。DB 迁移使用 SQLite `ALTER TABLE ADD COLUMN` + `PRAGMA table_info` 检测，实现**幂等**迁移（已存在则跳过），避免重启时报错。

```python
# 市场 → 货币映射
_MARKET_CASH_FIELD = {
    "A": ("cash_cnh", "init_cnh"),
    "HK": ("cash_hkd", "init_hkd"),
    "US": ("cash_usd", "init_usd"),
}
```

### 6.2 K 线图技术栈选型

**为什么不用 ECharts？**：ECharts 体积 ~1MB，金融序列支持需手动配置 candlestick series，时间轴缩放体验一般。**Lightweight Charts（TradingView 出品）**：体积 40KB，开箱即用的专业蜡烛图 + 成交量组合，内置十字光标/价格线/时间轴缩放，是彭博/Robinhood 同级的 UI。通过 CDN 一行引入，无需打包依赖。

**后端 K 线接口设计**：`GET /api/v1/market/kline?symbol=AAPL&period=daily&count=120`
- A 股：akshare `stock_zh_a_hist`（前复权）主 + yfinance 备
- 港股/美股：yfinance（`ticker.history(period, interval)`）
- 统一返回 Lightweight Charts 原生格式：`[{"time":"2024-01-02","open":...,"close":...}]`
- 双线程熔断：内部 Thread + Event.wait(20s)，超时返回 `[]`

### 6.3 多平台热榜聚合架构

**为什么不让前端直接调用？**：微博/B站/知乎的接口有防盗链、referer 检测、频率限制。前端直接调用会被 CORS 拦截或触发反爬，且每个用户各自爬取会大量消耗第三方接口配额。后端统一缓存后，**N 次前端请求 = 1 次后端抓取**。

**15 分钟 TTL 缓存 + 后台刷新**：Server 启动后每 15 分钟用 `asyncio.create_task` 触发后台刷新，保证缓存常新而不阻塞事件循环。

**优雅降级**：5 个数据源各自独立 try/except，任何一个失败只影响该源，前端对空 items 直接隐藏卡片，用户无感知。

### 6.4 Dashboard 聚合接口设计（BFF 模式）

**动机**：Dashboard 页面原本需要发 3 个独立请求（账户数据 + 快讯 + 用户信息）。每次刷新产生 3 次网络往返，且任一失败都会导致局部白屏。

**解决方案**：`GET /api/v1/dashboard/summary` 一次性返回全部数据——这是**BFF（Backend For Frontend）**模式的微型实践，为特定 UI 聚合后端数据，减少前端请求次数，提升页面加载体验。
