## 研究意义

### 背景：大学生投资教育的数据困境

中国在校大学生 **2.9 亿**，其中有投资意愿但缺乏系统指导的比例持续上升

> "散户七亏二平一赚"——A 股长期规律

### 三大数据层痛点

- **数据获取难**：A 股数据接口对个人限流严重，港美股需要翻墙，学生根本拿不到完整行情
- **数据理解难**：原始财报/K 线对非金融专业学生毫无意义，需要 AI 翻译成"人话"
- **数据碎片化**：新闻在微博、行情在同花顺、研报在万得——没有一站式感知入口

### AI 产品经理视角：我们在解决什么问题

```
用户需求：一站式获取 + AI 解读 + 零门槛操作
技术翻译：多源数据感知 + LLM 语义理解 + 流式可视化呈现
```

**价值主张**：用大数据管道 + AI 感知能力，把"机构级数据基础设施"开放给每一个大学生

### 数据标准化管道

```
原始数据（各源格式不同）
    ↓ DataLoader._standardize()
统一格式 [timestamp, open, high, low, close, volume]
    ↓ TTL 缓存 300s
内存热数据（秒级响应）
    ↓ 过期后指数退避重试（3 次，1.5x base）
重新拉取
```

### akshare 列顺序陷阱（工程实战教训）

```
stock_zh_index_spot_em:  col[4]=涨跌幅%  col[5]=涨跌额
index_global_spot_em:    col[4]=涨跌额   col[5]=涨跌幅%  ← 顺序相反！
stock_financial_abstract_ths: 升序排列，数值为"862.28亿"字符串
```

**教训**：大数据管理的第一步不是存储，而是理解每个字段的真实含义

## 核心技术二：双服务器数据中继架构

### 为什么需要双服务器？

| 问题 | 单服务器方案 | 双服务器方案 |
| --- | --- | --- |
| A 股数据访问 | 香港服务器调 akshare → 延迟高/限流 | 内地服务器本地调用 → 秒级响应 |
| 港美股数据 | 内地服务器调 yfinance → 需翻墙 | 香港服务器直连 → 无障碍 |
| LLM API 调用 | 内地调阿里云百炼 → 正常 | 香港调百炼 → 同样正常 |
| 新闻源抓取 | 香港抓财联社 → 可能被封 | 内地抓国内源 → 原生访问 |

### 架构详图

```
用户浏览器 ──────────────── http://47.76.197.100
                                    │
                                    ▼
              ┌─────────────────────────────────────┐
              │  香港服务器 (47.76.197.100)           │
              │  Nginx 反向代理 + HTTPS               │
              │  FastAPI 主服务 + LangGraph Agent     │
              │  港美股: yfinance 直连                │
              │  LLM: 阿里云百炼 Qwen3.5-plus        │
              │  前端: 8 页 HTML 静态托管             │
              └──────────────┬──────────────────────┘
                             │ REST API
                             │ Bearer token 鉴权
                             │ 指数退避重试
                             ▼
              ┌─────────────────────────────────────┐
              │  内地服务器 (47.108.191.110:8001)     │
              │  FastAPI 数据中继服务                  │
              │  A股/港股: akshare 本地调用            │
              │  新闻源: 财联社/新浪/澎湃             │
              │  BM25 RAG 稀疏检索                    │
              │  MySQL 数据库（用户/帖子/交易记录）     │
              └─────────────────────────────────────┘
```

### 中继协议设计

- **鉴权**：每次请求携带 Bearer token，防止中继接口被滥用
- **超时**：单次请求 10s 超时，超时后降级为本地缓存数据
- **重试**：指数退避 3 次（1s → 1.5s → 2.25s），避免雪崩
- **缓存**：内地服务器本地 TTL 缓存，减少 akshare 调用频率

## 核心技术三：混合 RAG 知识库

### 为什么用混合检索？

| 检索方式 | 优势 | 劣势 |
| --- | --- | --- |
| Chroma 向量 | 语义相似度高 | 对精确术语匹配弱 |
| BM25 稀疏 | 精确关键词命中 | 无法理解同义词 |
| **混合 RAG** | **兼得两者优势** | 需要两套索引 |

### 知识库构建流程

```
data/docs/ 研报 PDF
    ↓ scripts/build_kb.py
    ├─→ DashScope Embedding → Chroma 向量库 (data/chroma_db/)
    └─→ jieba 分词 → BM25 索引 (data/bm25_index.pkl)

查询时:
    用户问题 → Chroma top-k + BM25 top-k → 去重合并 → LLM 生成回答
```

### 存储架构

| 存储层 | 技术 | 用途 | 持久化 |
| --- | --- | --- | --- |
| 向量库 | ChromaDB | 研报语义检索 | `data/chroma_db/` |
| 稀疏索引 | BM25 (pickle) | 研报关键词检索 | `data/bm25_index.pkl` |
| 关系数据库 | MySQL (asyncmy) | 用户/帖子/交易记录 | 内地服务器 |
| 内存缓存 | Python dict + TTL | 行情热数据 | 进程内（重启丢失） |

## 核心技术四：SSE 流式数据传输

### 为什么不用 WebSocket？

| 维度 | WebSocket | SSE |
| --- | --- | --- |
| 方向 | 双向 | 服务端 → 客户端 |
| 复杂度 | 高（心跳/重连） | 低（HTTP 原生） |
| 适用场景 | 聊天室 | **AI 分析进度推送** |
| Nginx 兼容 | 需额外配置 | 天然支持 |

### SSE 事件类型设计

```
用户点击「开始分析」
    │
    ├─ event: start          → "分析启动，连接 Agent 引擎"
    ├─ event: node_start     → "数据 Agent 正在获取行情..."
    ├─ event: node_complete  → "基本面分析完成 ✓"
    ├─ event: conflict       → "发现多空分歧，启动辩论"
    ├─ event: debate         → "第 1 轮辩论结果..."
    ├─ event: risk_check     → "风控审核中..."
    ├─ event: risk_retry     → "风控拒绝，调整仓位重试"
    ├─ event: trade_order    → "模拟交易指令生成"
    └─ event: complete       → 完整研报 JSON
```

前端通过 `EventSource` 监听，逐步点亮节点进度条，实现"AI 思考过程可视化"

## 产品展示 · 个股分析

> `trade.html` — 平台核心功能，LangGraph 全链路 AI 分析实时可视化

**线上地址**：http://47.76.197.100/trade.html

### 页面全景

| 区域 | 内容 | 数据源 | 感知技术 |
| --- | --- | --- | --- |
| 股票输入区 | 代码输入 + 市场自动识别（A/港/美） | 用户输入 | 智能分类感知 |
| Agent 进度条 | 数据→基本面→技术面→情绪→RAG→组合→风控 节点实时点亮 | SSE 事件流 | 流式状态感知 |
| 财务图表 | 近 5 年营收/净利润趋势柱状图 | 内地 relay (akshare) / yfinance | 财务数据感知 |
| 辩论过程 | 多空双方论点展示（≤2 轮） | LangGraph debate_node | 冲突感知与消解 |
| 风控审核 | 四重硬风控通过/拒绝/重试状态 | LangGraph risk_node | 风险感知 |
| 投资研报 | 论文式完整研报（Markdown 渲染） | 4 路 Agent 汇聚输出 | 多源融合理解 |

### 分析流程

```
用户输入股票代码
    ↓ POST /api/v1/analyze → SSE 长连接建立
    ↓
[数据 Agent]  ← akshare(内地relay) / yfinance(香港直连)
    ↓ 并行下发
┌──────────────────────────────────────────────┐
│ 基本面Agent     技术面Agent    情绪Agent   RAG检索  │
│ 商业模式/护城河   MACD/RSI     新闻情感   研报知识  │
│ 催化剂/同行对比   布林带/量价   政策解读   宏观政策  │
└──────────────────────────────────────────────┘
    ↓ 汇聚
[组合 Agent] → 发现分歧？
  ├─ 是 → [多空辩论 ≤2轮] → 重新决策
  └─ 否 → [风控 Agent] → 四重硬风控
              ├─ 拒绝 → 组合Agent修正（反馈重试 ≤2次）
              └─ 通过 → 生成论文式研报 + 模拟交易建议
```

### 研报输出结构

| 章节 | 内容 |
| --- | --- |
| 投资建议 | BUY/HOLD/SELL + 置信度 + 建议仓位 + 止损止盈 |
| 投资论点 | 2-3 句核心理由 |
| 商业模式 | 公司靠什么赚钱？收入结构如何？ |
| 护城河 | 品牌/技术/网络效应/规模效应 |
| 估值对比 | PE/PB/ROE + 同行横向对比 |
| 催化剂 | 未来 1-2 季度推动股价的事件 |
| 核心风险 | 什么会推翻投资论点 |
| 情景分析 | 乐观/悲观两种情景定量估算 |
| 学生建议 | 分批建仓、止损纪律、持仓周期建议 |

### 大数据管理亮点

- **多源数据融合**：单次分析同时调用 akshare 行情、yfinance 财报、财联社新闻、Chroma+BM25 研报知识库，四路数据并行汇聚
- **SSE 事件流驱动**：后端每完成一个 Agent 节点即推送 `node_complete` 事件，前端逐步渲染进度条，无需轮询
- **Pydantic 结构化输出**：LLM 原始输出经 `AnalystReport` / `RiskDecision` / `TradeOrder` 强约束解析，杜绝幻觉字段
- **ECharts 可视化**：营收/净利润数据从 `stock_financial_abstract_ths`（带"亿"字符串）清洗为浮点数后渲染图表
- **市场差异化权重**：A 股(基本面40%/技术25%/情绪35%) vs 港股(55%/20%/25%) vs 美股(50%/25%/25%)，反映不同市场数据特征

### 产品经理思考

> 个股分析页是整个平台的"大脑"——它回答用户最核心的问题："这只股票值不值得买？"
> 从产品设计角度，关键决策有两个：
> 1. **过程透明**：不是给一个黑箱结论，而是让用户看到每个 Agent 在分析什么、辩论了什么、风控为什么通过/拒绝。这是教育平台和炒股软件的本质区别。
> 2. **结构化研报**：输出不是一段散文，而是论文式结构（论点→证据→风险→建议），培养用户系统性思维。

### 大数据管理亮点

### 产品经理思考

> 模拟交易页是"感知 → 理解 → 行动"的完整闭环。
> 从产品角度，关键决策是：**分析结果直接联动交易面板**——用户看完研报，一键就能模拟下单。
> 这不是两个割裂的功能，而是一条完整的用户旅程。

### 风控硬约束（代码层强制）


## 工程实践与反思

### 大数据管道的真实挑战

**挑战一：数据源不可靠**

akshare 接口不定期变更字段名、列顺序，甚至返回空数据。应对：多源备用 + 降级策略 + 指数退避重试。

**挑战二：跨境数据访问**

香港服务器无法稳定访问 A 股数据，内地服务器无法访问 yfinance。应对：双服务器中继架构。

**挑战三：数据"脏"程度超出预期**

`stock_financial_abstract_ths` 返回的数值是"862.28亿"这样的字符串，需要手动解析单位。各接口列名不统一（有的叫"收盘价"，有的叫"close"）。

### AI 产品经理的核心认知

> 做 AI 产品最难的不是模型，而是 **数据** 和 **工程**。

```
AI 产品的能力上限 = 数据质量 × 工程健壮性 × 模型能力
```

三者缺一不可。课程告诉我们"智能感知"和"大数据管理"——这个项目让我体会到：

- **感知**的难点不在算法，在于数据源的不稳定和异构性
- **管理**的难点不在存储，在于跨境数据流动和缓存一致性
- **产品**的难点不在功能，在于让用户对底层复杂度无感知

### 如果继续迭代

1. **数据血缘追踪**：记录每条数据从哪个源、什么时间、经过哪些清洗步骤到达用户
2. **智能预取**：基于用户行为预测下一步查询的股票，提前缓存数据
3. **社区数据反哺**：用社区讨论热度作为情绪 Agent 的补充信号源
4. **数据质量监控**：自动检测数据源异常（空值率、延迟、格式变更），触发告警
